STEP 5 — RLS policies
──────────────────────────────────────

  Invoke skill: /supabase-postgres-best-practices  (read before writing any SQL)

  Create supabase/migrations/001_rls_policies.sql with RLS policies for all 8 tables.
  Base them on the rules in CLAUDE.md § 5 and Namo_Product_Spec_MVP.md § 9.2.

  Mandatory policies to implement:

  users table:
    - Enable RLS
    - SELECT: own row for volontario; all rows for super_admin
    - UPDATE: own non-sensitive fields for volontario; all for super_admin
    - phone_encrypted column: SELECT restricted to super_admin only
      (use a column-level security policy or a separate view)
    - INSERT: only via service role (registration is handled server-side)
    - DELETE: only via service role

  user_tags table:
    - SELECT: authenticated users can read all tags
    - INSERT/UPDATE/DELETE: super_admin only

  user_tag_assignments table:
    - SELECT: own assignments for volontario; all for super_admin
    - INSERT/DELETE: super_admin only

  events table:
    - SELECT: published+interno OR published+aperto for authenticated users;
              published+aperto only for unauthenticated (anon);
              all rows for super_admin
    - INSERT/UPDATE/DELETE: super_admin only

  registrations table:
    - SELECT: own rows for volontario; all rows for super_admin
    - INSERT: own rows for volontario (capacity check handled at app layer)
    - UPDATE: own rows for volontario (status transitions only); all for super_admin
    - DELETE: no one — use status='cancelled' instead of DELETE

  external_registrations table:
    - SELECT: super_admin only (no self-service view for external users)
    - INSERT: anon (public) — for registration
    - UPDATE: super_admin only (cancellation via tokenised route done server-side)
    - DELETE: super_admin only (GDPR deletion)

  audit_log table:
    - SELECT: super_admin only
    - INSERT: authenticated users (app writes via server action)
    - UPDATE: nobody
    - DELETE: nobody

  notification_preferences table:
    - SELECT/UPDATE: own row for volontario; all for super_admin
    - INSERT: own row for volontario (created on account approval)

  Apply the migration to local Supabase after writing it.
  Test each policy using the Supabase SQL editor with different role contexts.

CONSTRAINTS AND GOTCHAS
────────────────────────────────────────────────────────────────────────────────

  1. NEVER use the old @supabase/auth-helpers-nextjs package.
     Use @supabase/ssr exclusively (the nextjs-supabase-auth skill covers this).

  2. Use Server Actions for all form submissions and mutations.
     Do NOT create /api/ routes for actions that have an authenticated user context.

  3. The `users` table in Drizzle is separate from Supabase Auth's auth.users table.
     When a new user registers via Supabase Auth (or self-registration form), you MUST
     also create a row in the public.users table (via a Postgres trigger or a
     server-side insert after auth signup). Use a Postgres trigger for this:
       CREATE OR REPLACE FUNCTION handle_new_auth_user()
       RETURNS trigger AS $$
       BEGIN
         INSERT INTO public.users (id, email, status, role)
         VALUES (NEW.id, NEW.email, 'pending', 'volontario');
         RETURN NEW;
       END;
       $$ LANGUAGE plpgsql SECURITY DEFINER;
       CREATE TRIGGER on_auth_user_created
         AFTER INSERT ON auth.users
         FOR EACH ROW EXECUTE FUNCTION handle_new_auth_user();
     Include this trigger in the Supabase migration SQL.

  4. The volunteer registration form (Step 6 /registrati) collects more fields than
     Supabase Auth knows about (name, phone, sectors, etc). The flow is:
       a. Create Supabase Auth user (email+password via signUp())
       b. The trigger above creates the public.users row with status=pending
       c. Immediately after, run a server action to update the public.users row
          with the additional fields (name, phone, sectors, nickname, notes)
     Do NOT try to store extra fields in Supabase Auth metadata — use public.users.

  5. Google OAuth users also go through the same trigger + status=pending flow.
     On first OAuth login, the user will be redirected to /in-attesa.
     A coordinator must approve them before they can access the app.
     The /auth/callback route handler must handle this redirect correctly.

  6. Do NOT implement any payment, subscription, or billing logic.
     Do NOT implement the Sector Admin role — it is explicitly out of scope.

  7. All UI strings must be in Italian. If you are unsure of the correct Italian
     phrasing, write a comment: // TODO: Italian copy needed here
     Never leave English strings silently in production UI.

  8. Do not add any features, components, or pages beyond what is listed in this
     prompt. Placeholder pages for future milestones are fine — full implementations
     are not. Keep scope tight.

  9. When writing the schema, do NOT store waitlist_position as a column in the
     registrations table. Waitlist position is always computed at query time as:
       SELECT COUNT(*) FROM registrations
       WHERE event_id = $1 AND status = 'waitlist' AND registered_at < $2
     This is documented in CLAUDE.md § 6 and ADR-06 in the spec.

 10. The audit_log table must have NO update or DELETE RLS policy and no endpoint
     that allows modification. It is append-only by design. If you accidentally
     write any update/delete path for audit_log, remove it.
